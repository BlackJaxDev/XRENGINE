# Audio2Face-3D Component

`Audio2Face3DComponent` is a scene component that drives facial blendshapes from the `animation_frames.csv` output produced by NVIDIA Audio2Face-3D tooling, supports optional Audio2Emotion curves in the same component, and exposes a live-stream runtime path for externally registered adapters.

For the full machine setup and native bridge build flow, see [Audio2Face-3D Engine Setup](audio2face-3d-engine-setup.md).

## What It Does

- Reuses the same scene-node pattern as `OVRLipSyncComponent`: place it next to an `AudioSourceComponent` and a `ModelComponent`.
- Starts animation playback when the sibling audio source begins streaming audio.
- Samples the exported Audio2Face-3D keyframes over time and applies the returned blendshape weights to the sibling model.
- Accepts Audio2Emotion channels (`angry`, `disgust`, `fear`, `happy`, `neutral`, `sad`) from CSV or live adapters and maps them onto configurable blendshape targets.
- Supports blendshape name prefix/suffix remapping when your mesh naming differs from the CSV.
- Supports project-relative CSV paths and an editor inspector with browse, reload, and import-to-project actions.
- Exposes a live-stream source mode so a native NVIDIA Audio2X bridge can push blendshape and emotion frames into the component without another runtime refactor.
- Includes an `Audio2Face3DNativeBridgeComponent` that can subscribe directly to `MicrophoneComponent.BufferReceived`, normalize microphone PCM into the bridge input format, and feed live inference once the native shim DLL is available.

## Expected Input

Point `AnimationCsvPath` at a CSV generated by the NVIDIA samples. The header can contain blendshape columns, Audio2Emotion columns, or both. A typical combined header looks like this:

```text
timecode,EyeBlinkLeft,EyeBlinkRight,JawOpen,happy,sad,...
```

Each subsequent row should contain a frame time followed by normalized weights.

## Current Scope

The shipped runtime includes two source modes:

- `CsvPlayback` for `animation_frames.csv` playback.
- `LiveStream` for a registered live client adapter.

XREngine now includes the managed side of the native bridge and a starter shim project at `Build/Native/Audio2XBridge/Audio2XBridge.vcxproj`.

The repo also now includes a dependency getter for the upstream NVIDIA SDK source:

- `Tools/Dependencies/Get-Audio2Face3DSdk.ps1`
- `Tools/Dependencies/Get-TensorRT.ps1`
- `Tools/Dependencies/Get-CudaToolkit.ps1`
- VS Code task: `Install-Audio2Face3D-SDK`
- VS Code task: `Install-TensorRT`
- VS Code task: `Install-CUDA`
- VS Code task: `Install-Audio2Face3D-Prereqs`

- The managed bridge is implemented by `Audio2Face3DNativeBridgeComponent`.
- The native shim uses a flat C ABI so the .NET runtime never talks directly to NVIDIA's C++ interfaces.
- The checked-in C++ implementation now contains the real bridge path behind an SDK-enabled build flag; when the upstream SDK build products are missing it still compiles in fallback mode and returns clear backend-unavailable errors.

That split keeps the editor build stable while still giving the runtime a real bridge boundary:

- `Audio2Face3DNativeBridgeComponent` owns the live session.
- `MicrophoneComponent` provides raw mono PCM.
- `Audio2Face3DNativeBridgeAudioConverter` converts 8-bit, 16-bit, or 32-bit microphone buffers into mono 16-bit PCM at the bridge sample rate.
- The native shim is expected to buffer audio, run NVIDIA inference, and expose the latest blendshape/emotion frames through the exported polling API.

## Typical Setup

1. Generate `animation_frames.csv` with the NVIDIA Audio2Face-3D sample workflow.
2. Add `Audio2Face3DComponent` to the same node as your `AudioSourceComponent` and `ModelComponent`.
3. Set `AnimationCsvPath` to the exported CSV.
4. Play or stream the matching audio through the sibling audio source.

For live inference with the native bridge:

1. Add `Audio2Face3DComponent`, `Audio2Face3DNativeBridgeComponent`, and `MicrophoneComponent` to the same scene node.
2. Acquire prerequisites:
	- Run `pwsh Tools/Dependencies/Get-CudaToolkit.ps1` or the `Install-CUDA` task to stage a CUDA installer. Supply `-DownloadUrl` for unattended download or `-InstallerPath` to reuse a local installer. Add `-Install` if you want the script to launch the installer silently.
	- Run `pwsh Tools/Dependencies/Get-TensorRT.ps1` or the `Install-TensorRT` task to stage and extract the TensorRT Windows SDK. The script now includes a built-in pinned mapping for TensorRT `10.13.3.9` on Windows CUDA `12.9`; if NVIDIA blocks the direct download, rerun with `-ArchivePath` after downloading the zip manually. Add `-SetUserEnvironment` if you want the script to persist `TENSORRT_ROOT_DIR`.
3. Run `pwsh Tools/Dependencies/Get-Audio2Face3DSdk.ps1` or the `Install-Audio2Face3D-SDK` task to clone/update the official NVIDIA SDK into `Build/Dependencies/Audio2Face-3D-SDK` and fetch its build dependencies.
4. If needed, set `CUDA_PATH` / `TENSORRT_ROOT_DIR` for the current shell, then rerun `Get-Audio2Face3DSdk.ps1 -Build`.
	- If you want a single entry point inside VS Code, run `Install-Audio2Face3D-Prereqs` to execute the CUDA, TensorRT, and Audio2Face SDK tasks in sequence.
5. Set `Audio2Face3DComponent.SourceMode` to `LiveStream`.
6. Build `Build/Native/Audio2XBridge/Audio2XBridge.vcxproj`. When the upstream SDK output exists at `Build/Dependencies/Audio2Face-3D-SDK/_build/release/audio2x-sdk`, the project automatically links the real backend and copies the required runtime DLLs into the bridge output folder.
7. If you want the bridge to own microphone setup, leave `AutoConfigureMicrophoneFormat` enabled so the microphone runs at 16 kHz PCM16 before capture starts.

If you also need NVIDIA's gated Hugging Face models, rerun the dependency script with `-DownloadModels` after authenticating `hf auth login`. Use `-GenerateTestData` if you want the SDK's TensorRT test assets as well.

In the editor, use the dedicated Audio2Face inspector to browse for a CSV, reload it, or import it into the active project's `Assets/Audio2Face/` folder.

## Notes

- NVIDIA sample output uses ARKit-style blendshape names; `ModelComponent.SetBlendShapeWeightNormalized` already matches names case-insensitively.
- If your avatar uses extra naming decoration, use `BlendshapeNamePrefix` and `BlendshapeNameSuffix`.
- The inspector exposes per-emotion target lists so the six Audio2Emotion channels can drive higher-level facial expression shapes on top of speech blendshapes.
- Live-mode adapters can call `MarkLiveClientConnected()`, `MarkLiveClientDisconnected(...)`, `TryUpdateLiveFrame(...)`, and `TryUpdateLiveEmotionFrame(...)` on the component from their own client implementation.
- The native microphone path is compatible with the bridge because `MicrophoneComponent` already emits mono PCM chunks through `BufferReceived`. The bridge component subscribes there directly rather than scraping playback buffers from `AudioSourceComponent`.
- `Audio2Face3DNativeBridgeComponent` now has a dedicated editor inspector for bridge status, SDK/model paths, microphone controls, and connect/disconnect actions.
- If the upstream Audio2X SDK has not been built yet, the bridge project still builds in fallback mode and `TryConnectLiveClient()` fails with a backend-unavailable error instead of silently pretending inference is active.
